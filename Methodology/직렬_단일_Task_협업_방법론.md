# 직렬 단일 Task 협업 방법론

**작성일**: 2026-05-26
**출처**: Lobster AI Music Web DAW 프로젝트 (사용자 + Claude + Antigravity 협업)
**용도**: 사람 + LLM(Claude·GPT·Gemini 등) + 코드 실행 에이전트(Antigravity·Cursor·Aider 등) 3자 협업을 위한 재사용 가능한 패턴 모음

---

## 0. 한 줄 요약

작업을 단일 Task로 직렬화하고, 코드 에이전트의 자체 검증을 인정하지 않으며, "완료" 라벨을 의심하는 카테고리를 별도로 두는 협업 방식. 누락을 강제로 명시하게 하고, 결정 사항을 사용자가 한눈에 보는 표로 관리. 시행착오로 발견된 사실들을 부록에 시간 역순으로 누적.

---

## 1. 왜 이 방법론이 만들어졌나

LLM과 코드 실행 에이전트를 함께 쓸 때 반복적으로 마주치는 문제가 있다.

1. **에이전트의 거짓 통과 보고**. Puppeteer·자체 단위 테스트로 "통과"라고 보고하지만 실제 사용자 환경에서는 동작 안 함.
2. **누적된 변경의 추적 불가능**. 변경한 사실을 보고에서 누락하거나, git에 커밋 안 한 채로 며칠씩 잔존.
3. **여러 Task를 한 번에 던졌을 때의 축약**. 묶음 작업 중 일부를 임의로 생략하거나 합쳐서 처리.
4. **의심 항목과 확정 항목의 혼재**. "해결됨"으로 분류된 것이 실제로는 검증되지 않은 경우.
5. **결정 사항이 대화 흐름에 묻힘**. 사용자가 다음에 뭐 해야 하는지 매번 다시 찾아야 함.

이 5가지 문제를 구조적으로 차단하는 패턴을 아래에 정리한다.

---

## 2. 핵심 원칙 8가지

### 2.1 단일 Task 직렬 처리

한 번에 하나의 Task만 코드 에이전트에 전달한다. 묶어 던지면 누락·축약·생략 위험이 증가한다. 이전 Task가 사용자 실측 통과한 후에만 다음 Task의 작업 프롬프트를 작성·전달한다.

작은 작업이라도 의존성이 다른 결정을 포함하면 분리한다. 예: "정리 작업"이라도 "사용자 결정 불필요한 부분"과 "사용자 결정 필요한 부분"은 다른 Task.

### 2.2 자체 검증 = "통과" 아님

코드 에이전트의 Puppeteer 자체 검증, 단위 테스트 통과, 컴파일 성공, 타입 체크 통과는 "통과" 라벨을 받지 못한다. 사용자가 실제 환경에서 직접 확인한 결과 보고가 들어와야만 "통과" 처리한다.

이를 강제하기 위해 모든 작업 프롬프트 헤더에 다음 한 줄을 박는다.

> "Puppeteer 검증만으로 완료 단언 금지. 사용자 OOO 실제 환경 실측 보고 전까지 어떤 Task도 '완료' 처리 금지."

### 2.3 누락 명시 강제

코드 에이전트는 의도적으로든 비의도적으로든 일부 작업을 생략한다. 이를 차단하려면 보고 형식에 "누락 항목" 필드를 의무로 둔다.

```
- 누락 항목: (있다면 명시. 없으면 "없음"이라고 적을 것)
```

이 한 줄이 있으면 에이전트가 작업 중 막힌 부분을 솔직하게 보고한다. 없으면 자기 작업을 "완벽하게 완료했다"고 포장하기 쉽다.

### 2.4 의심 항목 격리 (R 카테고리)

"해결됨"으로 분류된 항목 중 의심스러운 것을 별도 카테고리(R: Re-verify)로 격리한다. R 카테고리는 "확정 미해결"이 아니라 "검증되지 않은 해결". 이걸 본문에 두면 검증 가능 환경이 복구된 후 실측 체크리스트로 일괄 처리할 수 있다.

R 카테고리 라벨 예시:
- R-1: 자체 검증 위반 사례 → 다음 Task에 흡수
- R-2: 변경 이력 추적 불가능 → 영구 보류 라벨링
- R-3: 과거 시기 완료 항목 전수 재검증 → 체크리스트 작성

### 2.5 작업 프롬프트 표준 헤더

모든 코드 에이전트 작업 프롬프트는 동일한 [원칙] 헤더를 박는다. 헤더의 항목은 프로젝트별로 조정하되, 핵심 5개는 공통.

1. 문체 통일 ("~합니다/했습니다" 또는 "~다/~이다" 등 일관)
2. 금지 표현 명시 ("완벽", "100%", "완전히" 등 과장 표현 차단)
3. 자체 검증만으로 완료 단언 금지
4. 누락 시 명시 강제
5. 임의 생략 금지

프로젝트 특성에 따라 추가:
- 인코딩 강제 (한글·다국어 프로젝트의 경우 UTF-8 BOM 없이 저장)
- 토큰화된 CSS 변수만 사용 (디자인 시스템 프로젝트)
- 상수 import 강제 (브랜딩 일관성)
- 패치 스크립트 인코딩 옵션 강제

### 2.6 보고 형식 표준화

코드 에이전트의 회신 형식을 표준화한다. 자유 형식 응답은 추적 어렵다. 다음 구조를 강제.

```
## Task <ID> 결과
- 진단 결과: (구체적 사실)
- 원인: (가능성 분석)
- 수정 파일·라인: (diff 또는 변경 위치)
- 에이전트 자체 검증: (Y/N - 코드 변경 확인 수준)
- 사용자 실측 결과: 보류 (사용자 보고 대기)
- 누락 항목: (있다면 명시, 없으면 "없음")
```

진단 Task인 경우 추가:
```
- 조사 항목 1: (객관 데이터)
- 조사 항목 2: (객관 데이터)
- 종합 권장 (1줄):
```

### 2.7 결정 표 사용자 친화화

사용자가 결정해야 할 항목은 표로 정리한다. 본문에 풀어쓰면 흐름에 묻혀 누락된다.

```
| 코드 | 결정 항목 | 현재 상태 | Claude 권장 |
|------|----------|----------|------------|
| B-1  | 항목 A    | 미답     | 옵션 X     |
| B-2  | 항목 B    | 미답     | 옵션 Y     |
```

각 결정에 코드(B-1, B-2 등)를 부여하고, 후속 답변에서 코드로 참조하면 추적 가능하다.

### 2.8 의심을 시스템화

사용자가 "이게 정말 해결됐는가" 의심하는 순간은 가장 가치 있는 순간이다. 의심을 무시하지 않고 R 카테고리로 즉시 분류한다.

의심 발생 → 격리 → 검증 가능 환경 복구 → 실측 체크리스트 → 일괄 처리 순서.

## 3. 핵심 카테고리 체계

작업 항목을 카테고리로 분류해 추적한다. 프로젝트 규모에 따라 조정.

| 카테고리 | 의미 | 처리 주체 |
|---------|------|----------|
| A | 다음 Task (코드 에이전트 작업) | 코드 에이전트 |
| B | 사용자 결정 대기 | 사용자 |
| R | 재검증 필요 (의심 항목) | 사용자 실측 |
| D | 큰 통합 작업 (사용자 비전 핵심) | 코드 에이전트 + LLM |
| E | 큰 별도 라운드 (디자인·로그·통계) | 코드 에이전트 + LLM |
| F | 큰 결정 (지금 결정 불필요) | 사용자 |

A·B·R은 즉시 처리 흐름. D·E·F는 큰 의사결정 후 큰 라운드로.

## 4. 문서 구조 표준 (핸드오프 문서)

여러 세션에 걸쳐 진행되는 프로젝트라면 핸드오프 문서를 만든다. 다음 구조를 권장.

```
표지: 프로젝트명 + 버전 + 작성일 + 변경 요약 1줄

§0 사용자 요약 시트 (1페이지)
   ├─ 0.1 지금 당장 할 일 (3단계)
   ├─ 0.2 결정 대기 (해당 Task 직전에만 답)
   ├─ 0.3 재검증 필요 (R 카테고리)
   └─ 0.4 큰 결정 결과 요약

§1 프로젝트 정체성
   ├─ 1.1 핵심 (사용자 비전 한 문단)
   ├─ 1.2 환경 + 보유 자산 (코드는 git이 진실, 한 줄 요약)
   └─ 1.3 협업 패턴 (역할 정의)

§2 미완료·미결정·재검증 마스터 리스트
   ├─ A 카테고리 (다음 Task)
   ├─ B 카테고리 (사용자 결정)
   ├─ R 카테고리 (재검증)
   ├─ D 카테고리 (큰 통합)
   ├─ E 카테고리 (큰 별도 라운드)
   └─ F 카테고리 (큰 결정)

§3 작업 순서 (직렬 단일 Task)
   └─ 단계별 표 (작업·담당·예상)

§4 큰 결정 사항 + 외부 의존성 + 제약
   └─ 옵션 비교표·검증 결과·알려진 한계

§5 코드 에이전트 작업 프롬프트 패턴
   ├─ 5.1 [원칙] 헤더
   ├─ 5.2 보고 형식 표준
   ├─ 5.3 컨텍스트 제공 워크플로우
   └─ 5.4 룰 위반 방지책

§6 즉시 실행 작업 프롬프트 (복사·붙여넣기용)

[부록 A] 진행 로그 (시간 역순)
   ├─ A.1 최근 세션
   └─ A.2 이전 누적 (재검증 대상은 R로 라벨)

[부록 B] 결정 근거 로그
   └─ 청취 검증·이미지 분석·옵션 비교·외부 의존성 작동 원리
```

### 4.1 본문 vs 부록 분리 원칙

- **본문**: 미완료·미결정·재검증 항목만
- **부록**: 완료된 작업 (시간 역순)

이렇게 분리하면 새 세션이 본문만 읽고도 다음 작업을 파악할 수 있다. 회귀 추적이 필요할 때만 부록을 참조한다.

### 4.2 R 카테고리 라벨링

- "해결됨"으로 분류된 항목이 의심스러우면 즉시 R로 옮긴다
- R-N에 처리 방향 명시 (영구 보류·다음 Task에 흡수·체크리스트 작성)
- R 카테고리는 "검증 안 됨"이지 "미해결"이 아님 — 코드 자산은 신뢰

## 5. 작업 흐름 (한 사이클)

한 Task 처리의 표준 사이클은 다음과 같다.

### 5.1 LLM이 작업 프롬프트 작성
- [원칙] 헤더 + 배경 + 작업 절차 + 수정 금지 + 보고 형식
- 사용자에게 "이 프롬프트를 코드 에이전트에 전달하세요" 안내
- 첫 Task는 핸드오프 문서 첨부 + "이 문서는 전체 컨텍스트. 모든 Task에서 §5 원칙 준수" 한 줄

### 5.2 사용자가 코드 에이전트에 전달
- 작업 프롬프트 복사·붙여넣기
- 컨텍스트 문서 첨부 (첫 Task만)

### 5.3 코드 에이전트가 작업·보고
- 표준 보고 형식으로 회신
- 사용자 실측 결과는 "보류" 라벨

### 5.4 사용자가 LLM에 회신 전달
- 코드 에이전트 회신을 LLM 채팅에 붙여넣기

### 5.5 LLM이 회신 평가
- §5 룰 위반 여부 확인
- 새 발견 사항 추출 (의도 외 변경·누락 항목 등)
- 사용자 실측 시나리오 작성 (구체적 명령어 또는 조작 단계)

### 5.6 사용자 실측
- LLM이 제시한 시나리오 그대로 실행
- 결과를 LLM에 회신

### 5.7 LLM이 통과·실패 라벨링
- 통과면 다음 Task의 작업 프롬프트 작성
- 실패면 회귀 진단 Task 작성
- 핸드오프 문서 갱신 (부록 진행 로그에 항목 추가)

---

## 6. §5 룰 위반 식별 패턴

LLM이 코드 에이전트 회신을 평가할 때 §5 룰 위반을 명시적으로 식별한다.

### 6.1 자체 검증만으로 완료 단언

징후:
- "정상 작동을 확인하였습니다"
- "이상 없이 통과되었습니다"
- 사용자 실측 결과 없이 "완료" 라벨

조치: 사용자 실측 시나리오 제시. 보류 라벨로 되돌림.

### 6.2 누락 미명시

징후:
- "누락 항목: 없음"이지만 후속 진단에서 누락 발견
- 의도 외 변경이 보고에 안 나타남
- working tree에 며칠씩 미커밋 변경 잔존

조치: R-N 신규 추가. 다음 작업 프롬프트에 누락 명시 강조.

### 6.3 임의 축약·생략

징후:
- 작업 절차 중 일부 단계만 실행
- "유사한 작업이라 묶어서 처리"
- 보고에서 단계 번호 건너뜀

조치: 직렬 단일 Task 원칙 재확인. 작업 프롬프트 더 작은 단위로 쪼갬.

### 6.4 추측 기반 결론

징후:
- "추정됩니다"·"~일 것으로 보입니다" 같은 표현이 객관 근거 없이 사용
- git log·파일 mtime 등 근거 제시 없이 결론

조치: 작업 프롬프트에 "객관 근거 제시" 한 줄 추가.

## 7. 의심 해소 워크플로우

사용자가 "정말 해결됐는가" 의심하는 순간의 처리.

### 7.1 의심 발생
사용자가 명시적으로 의심을 표현 → 그 자체가 가장 가치 있는 신호.

### 7.2 즉시 R 카테고리 격리
- 해당 항목을 본문 "완료"에서 빼고 R-N으로 이동
- 이유·근거·처리 방향 1줄씩 기록

### 7.3 검증 가능 환경 복구
- git untracked 같은 추적 불가 환경 우선 복구
- 의심 항목 외 추가 발견 있을 가능성 — 발견 시 다시 R에 추가

### 7.4 실측 체크리스트 작성
- LLM이 5~10분 분량의 사용자 실측 항목으로 압축
- 각 항목은 구체적 명령어 또는 클릭 시퀀스
- 실측 결과는 통과·실패로 이분 라벨

### 7.5 일괄 처리
- 실패 항목만 다음 Task로 진행
- 통과 항목은 R에서 부록 진행 로그로 이동

---

## 8. 다양한 LLM·에이전트에 적용

이 방법론은 특정 LLM·에이전트에 종속되지 않는다.

### 8.1 LLM (Claude·GPT·Gemini 등)
- 첫 메시지에 이 문서 첨부
- "이 방법론을 따라 협업을 진행합니다" 한 줄
- 프로젝트 핸드오프 문서가 있으면 같이 첨부

### 8.2 코드 실행 에이전트 (Antigravity·Cursor·Aider·Cline 등)
- LLM이 작성한 작업 프롬프트를 그대로 전달
- 컨텍스트가 유지 안 되는 에이전트는 매 Task마다 핸드오프 문서 첨부
- 보고 형식 표준은 에이전트 종류와 무관하게 동일

### 8.3 단일 LLM 단독 (코드 에이전트 없이)
- LLM이 코드 작성 + 사용자가 직접 적용
- "사용자 실측 보고"는 사용자가 직접 환경에서 확인하는 것
- 자체 검증 vs 사용자 실측 구분은 동일하게 유지

### 8.4 프로젝트 종류별 조정
- 코드 프로젝트: 위 패턴 그대로
- 문서·기획 프로젝트: A·B·R·D·E·F 카테고리는 유지, "사용자 실측"을 "사용자 검토"로 치환
- 디자인 프로젝트: 작업 프롬프트의 토큰화·CSS 변수 항목 강화

## 9. 안티패턴 (피해야 할 것)

이 방법론을 적용해도 다음 행동은 효과를 무력화한다.

- **묶어 던지기**. "이것도 해주고 저것도 해줘" — 단일 Task 원칙 위반
- **자체 검증 인정**. "에이전트가 통과라고 했으니 됐다" — 사용자 실측 없이 통과 라벨 부여
- **의심 무시**. "찜찜하지만 그냥 넘어가자" — R 카테고리 격리 안 함
- **부록과 본문 혼재**. 완료 항목을 본문에 두면 추적 흐려짐
- **결정 흐름에 묻기**. 사용자 결정을 표 아닌 본문에 풀어쓰기
- **추측을 사실처럼 보고**. 객관 근거 없이 결론
- **누락 안 적기**. 보고에 "누락 항목: 없음"으로만 표시하고 검증 안 함

---

## 10. 적용 체크리스트

새 프로젝트에 이 방법론을 적용할 때 다음 7개를 점검한다.

1. 핸드오프 문서 §0 사용자 요약 시트 1페이지 작성됨
2. 카테고리 A·B·R·D·E·F 모든 항목 분류됨
3. 모든 작업 프롬프트에 [원칙] 헤더 + 보고 형식 표준 박혀있음
4. 의심 항목이 본문 "완료"가 아닌 R 카테고리에 격리됨
5. 본문에는 미완료만, 부록에 진행 로그가 시간 역순으로 누적됨
6. 결정 사항이 표로 정리됨 (코드 부여, 사용자가 코드로 답변)
7. 사용자 실측 시나리오가 구체적 명령어·조작 단계로 제시됨

7개 다 통과하면 이 방법론이 정상 적용된 것이다.

## 11. 시작 템플릿

새 LLM 채팅에 이 방법론을 적용할 때 첫 메시지 템플릿:

```
[프로젝트 명: OOO]
[작업: OOO]

협업 방법론으로 "직렬 단일 Task" 패턴(첨부 문서)을 사용합니다.

이 방법론의 핵심:
1. 단일 Task 직렬 처리 (한 번에 하나만)
2. 에이전트 자체 검증 = "통과" 아님 (사용자 실측만 통과)
3. 누락 명시 강제 (보고 형식)
4. 의심 항목은 R 카테고리로 격리
5. 카테고리 A·B·R·D·E·F 체계
6. 본문 = 미완료, 부록 = 완료 로그 (시간 역순)
7. 결정은 표로 정리, 코드 부여

당신은 LLM 역할 (설계·검증·작업 프롬프트 작성).
저는 사용자 역할 (결정·실측).
[코드 에이전트 사용 시: OOO이 코드 실행 역할].

[핸드오프 문서 있으면 첨부]

먼저 [현재 작업 상황 요약]을 알려드립니다.
[현재 상황]
```

이 템플릿으로 시작하면 새 LLM도 즉시 이 방법론을 따라 협업한다.

## 부록: 빠른 참조

### 카테고리 코드 체계
- A-N: 다음 Task (Antigravity 등 코드 에이전트 작업)
- B-N: 사용자 결정 대기
- R-N: 재검증 필요 (의심 항목 격리)
- D-N: 큰 통합 작업
- E-N: 큰 별도 라운드
- F-N: 큰 결정 (지금 결정 불필요)

### 보고 형식 표준 (복사용)
```
## Task <ID> 결과
- 진단 결과:
- 원인:
- 수정 파일·라인:
- 에이전트 자체 검증: Y / N
- 사용자 실측 결과: 보류
- 누락 항목:
```

### [원칙] 헤더 (복사용, 프로젝트별 조정)
```
[원칙]
- "~합니다/했습니다" 문체. 사실 위주, 형용사 최소.
- 금지 표현: "완벽", "완전히", "100%", "원천", "수려", "정화",
  "매직", "완성도" 일체.
- 자체 검증만으로 완료 단언 금지. 사용자 실측 보고 전까지
  어떤 Task도 "완료" 처리 금지.
- 누락 시 "OO은 누락" 명시. 임의 생략 금지.
- 추측 금지. 객관 근거(파일 내용·로그·시점 등) 제시.
- [프로젝트 특화 룰 추가]
```
